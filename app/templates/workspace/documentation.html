Getting Started

1. Log in & choose a Workspace

After logging in you land on the Workspace selector. Pick the workspace you want to explore, or ask your administrator to invite you to one.

2. Explore the Workspace tree

The main view is the Workspace Grid — a collapsible tree that shows the full CISK hierarchy. Use the three detail levels to control how much information is displayed:

  • Minimal — entity names only, cleanest view
  • Standard — names plus KPI values and consensus indicators
  • Full Detail — everything including impact levels, targets, and governance badges

3. Contribute values

Click any KPI cell to open the contribution panel. Enter your value (numeric, risk level, or list choice), add an optional comment, and submit. Your entry joins the consensus calculation automatically.

4. Use Dashboards for insight

Navigate to Dashboards in the top menu to access KPI performance, challenge execution, system health, impact coverage, and more.

Tip: Use the global search bar (top of page) to quickly find any entity. Type @requires_action or @incomplete to filter by status.

Workspaces

What is a Workspace?

A Workspace is a self-contained environment for a specific organisation or project. Each workspace has its own hierarchy of spaces, challenges, initiatives, systems, and KPIs. Users can belong to multiple workspaces with different permissions in each.

Switching Workspaces

Click the Workspaces dropdown in the navigation bar and select your target workspace. Your tree state, filters, and presets are saved per workspace.

Portal Systems

A system can be linked to another workspace, creating a Portal. Portal systems appear with a gradient pill in the tree showing the linked workspace's logo — click it to navigate across.

Contributions

How data enters the system

Values in CISK are entered through contributions. Each contributor provides their independent assessment for a KPI cell. The platform then calculates consensus from all contributions.

  • Numeric values — costs, counts, percentages (e.g., 150,000)
  • Qualitative levels — risk, impact, sentiment on a 1-3 scale
  • List choices — select from a predefined set of options
Contributors are identified by name and don't need user accounts. This makes it easy to collect expert opinions from outside the platform.

Consensus Calculation

How consensus is determined

When multiple contributors provide values for the same KPI cell, CISK computes a consensus status to determine the cell's value and whether it can participate in roll-ups.

StatusConditionRoll-up Eligible
No Data No contributions yet No
Strong Consensus Single contribution, or all contributors agree Yes
Weak Consensus Majority (> 50%) but not unanimous No
No Consensus No clear majority among contributors No
Important: Only values in Strong Consensus participate in upward roll-ups. This ensures aggregated numbers are based on agreed-upon data.

Aggregation (Roll-Up)

How values flow upward

Values are aggregated from the bottom of the hierarchy to the top:

KPI System Initiative Challenge Space

At each level, a configurable aggregation formula combines child values into a parent value.

Available aggregation formulas

FormulaBehaviourBest For
SUMTotal of all child valuesCosts, emissions, counts
AVGArithmetic meanSentiment, satisfaction
MAXHighest value winsRisk, impact (worst case)
MINLowest valueBest-case scenarios
MEDIANMiddle value (robust to outliers)General-purpose
COUNTNumber of contributing children"How many" metrics
MODEMost frequent valueList-type value types

Rollup Rules

Administrators can configure Rollup Rules to override the default aggregation formula at any link in the hierarchy, or disable roll-up entirely for specific branches.

Qualitative types (risk, impact, level, sentiment) cannot use SUM — it is automatically converted to AVG for these types.

Formulas & Calculated Values

Value Type Formulas

A value type can be defined as a formula that computes its value from other value types on the same KPI. Two modes are available:

Simple Mode

Choose an operation and source value types:

add: sum of all source values
subtract: first value − sum of remaining
multiply: product of all values
divide: first value ÷ product of remaining

Advanced Mode

Write a Python-safe expression referencing source value types by name:

round((revenue - cost) / cost * 100)   # ROI %
(kpi_actual / kpi_target) * 100        # Achievement %
(kpi_1 * 0.7 + kpi_2 * 0.3)           # Weighted average

Available functions: abs(), round(), max(), min(), sum()

KPI Value Priority Chain

When determining a KPI cell's value, CISK checks these sources in order:

  1. Value Type Formula — computed from other value types on the same KPI
  2. KPI Formula — computed from other KPIs at the same cell
  3. Linked KPI — read from a source in another workspace
  4. Manual Contributions — consensus from contributor entries

Impact Compounding Methods

True Importance

Each entity has an assessed impact level (1-3) representing its importance to its parent. The true importance is computed by compounding these levels through the full chain from root to leaf.

Space ★★ × Challenge ★★★ × Initiative ★ = True Importance ?

Method 1: Simple Product

Basic

Multiply weights through the chain, normalise by maximum possible score.

score = wspace × wchallenge × winitiative × ...
percentage = score / max_weightdepth
≤ 33% → Level 1  |  ≤ 66% → Level 2  |  > 66% → Level 3

Method 2: Geometric Mean

Balanced

At each step, compute the square root of parent weight × child weight. Stays naturally in the weight scale.

current = √(parent_weight × child_weight)
Final value mapped to nearest level (1, 2, or 3)

Method 3: Toyota QFD

Strongest Contrast

Uses a fuzzy lookup table — a weak link pulls the result down hard.

Child 3Child 2Child 1Child 0
Parent 33320
Parent 23210
Parent 12100

Methods 4 & 5: Toyota Weighted

Advanced

Applies exponential reinforcement: Level 1 → 1, Level 2 → 2, Level 3 → 4 (configurable).

  • Downstream Only — exponential reinforcement on the deepest (child) rating only
  • Full — exponential reinforcement on all levels in the chain
Normalised against max_rdepth → mapped to Level 1 / 2 / 3

⚠ No Consensus

Special State

When assessors cannot agree on an entity's impact level, select "No consensus" instead of a star rating. A disagreement note is required to document who disagreed and why.

  • No consensus breaks the compounding chain — the entity and all descendants show no True Importance (—)
  • A ⚠ warning badge appears in the workspace tree, making unresolved disagreements visible
  • Click the badge to read the note or change the assessment
  • The badge color is configurable in Workspace Administration → Impact Scale → Special State Colors
For detailed examples with real numbers, see Impact Calculation Docs.

Decision Register

The Decision Register tracks all decisions made within a workspace. Each decision includes:

  • What — the decision description
  • Who — who made the decision
  • Tags — scope, budget, timeline, resource, technical, governance
  • RAG status — traffic-light indicator
  • Entity mentions — link decisions to any CISK entity with @"Entity Name"
  • Governance Body — assign to an oversight entity

Action Items

The Action Register manages tasks and memos across the workspace. Features include:

  • Status tracking (open, in progress, done, blocked)
  • Priority levels and due dates
  • Governance body linking
  • Entity mentions with @"Entity Name"
  • Timeline view for visual planning
  • Duplicate detection using similarity analysis
  • JSON export/import with templates

Workspace Labels & Profiles

Organise your workspaces

When you have many workspaces, labels and profiles help you stay focused.

Labels

  • Labels are personal — each user creates and manages their own
  • Create labels on the Profile page → Workspaces tab
  • Assign labels to workspaces by clicking label badges on each workspace card
  • Click a label name to rename it, or the colour swatch to change its colour
  • Labels are linked by ID, so renaming is safe

Profiles

  • Profiles filter which workspaces appear in the Workspaces menu
  • Each profile selects a subset of labels — only workspaces with those labels are shown
  • Create profiles on the Profile page → Profiles tab
  • Activate a profile to apply its filter, Deactivate to show all workspaces
  • Profiles can also set Public Only mode to hide private spaces
  • Only one profile can be active at a time

  • Show CISK Systems — expand only branches leading to portal systems linked to other CISKs

Spaces

Value Type Configuration

Admin

Creating value types

Value types define columns in the workspace grid. Each value type has:

  • Kind — numeric, risk, positive/negative impact, level, sentiment, or list
  • Aggregation formula — how child values combine (SUM, AVG, MAX, etc.)
  • Display settings — unit, prefix/suffix, decimal places, divisor for scaling
  • Colour scale — min/mid/max colours for numeric heatmapping
  • Formula — optional computed value from other value types

List-type options

For list value types, define custom choices with a key, label, and colour. The aggregation formula is always MODE (most common choice). Ties return no value.

Rollup & Cache

Admin

Pre-computed rollups

When enabled, CISK pre-computes all aggregated values and stores them in a cache. This dramatically speeds up workspace loading (1 query instead of hundreds).

  • Cache is auto-invalidated when data changes (contributions, entity edits)
  • Stale cache is auto-recomputed on next workspace load
  • Manual recompute available via Administration or API

Rollup rules

Override the default aggregation formula at any link in the hierarchy. You can also disable roll-up entirely for specific branches — useful when a system or initiative should not contribute to its parent's aggregate.

Backup & Restore

Admin

YAML export / import

Export the full workspace structure as a YAML file for safe-keeping or migration. On import, you can map users and governance bodies from the source to the target workspace.

  • Includes all entities, KPIs, value types, contributions, decisions, and action items
  • Format version 3.0 with stable cross-reference IDs
  • Organisation cloning available for creating test environments
